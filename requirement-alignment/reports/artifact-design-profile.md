# Artifact Design Profile — 需求对齐摘要

> 定义「需求对齐摘要」工件的设计规范：结构、内容要求、质量标准。

## 工件定位

- **类型**：内联 Markdown 摘要（不写文件，直接输出到对话）
- **受众**：用户本人，作为 Plan 模式或执行阶段的输入
- **生命周期**：一次性使用，对齐完成后交给后续阶段

## 模板结构

```markdown
## 需求对齐摘要

### 真实目标
- <用户最终希望解决的问题或获得的结果>

### 当前任务
- <本次需要完成的具体工作>

### 范围
- 包含：<本次处理的内容>
- 不包含：<明确排除的内容>

### 必须保持
- <接口、行为、兼容性、数据格式、测试框架等约束>

### 验收标准
- <可观察、可验证的完成条件>

### 已确认决策
- <用户确认的关键选择>

### Agent 假设
- <仍未明确但允许继续使用的假设；没有则写"无">

### 后续入口
- <进入 Plan 模式 / 直接执行 / 继续只读调查>
```

## 内容要求

| 章节 | 要求 |
|------|------|
| 真实目标 | 用用户的语言，1~3 句，不添加用户未要求的新目标 |
| 当前任务 | 具体、可执行，不含技术方案细节 |
| 范围 | 明确包含和不包含，避免范围蔓延 |
| 必须保持 | 列出所有硬约束（接口、协议、行为、兼容性） |
| 验收标准 | 可观察、可验证，尽量量化（如"仿真通过""零丢帧""上板 24h 稳定"） |
| 已确认决策 | 记录用户确认的关键选择，可追溯 |
| Agent 假设 | 明确标注，区分于用户确认；无假设写"无" |
| 后续入口 | 明确下一步动作（Plan / 执行 / 只读调查） |

## 质量标准

- **使用用户的术语** — 不替换为用户不熟悉的表述
- **区分确认与假设** — 用户确认 vs Agent 假设必须显式区分
- **验收可执行** — 验收标准应能直接用于验证，不含模糊表述
- **长度匹配复杂度** — 简单任务 5~8 行，复杂任务 15~25 行，不追求形式完整
- **不添加新目标** — 只对齐，不扩展

## 反例

❌ 验收标准："性能提升" → 不可验证
✅ 验收标准："在 testbench 中，吞吐率从 800Mbps 提升到 1Gbps，且功能行为不变"

❌ Agent 假设："用户希望不改变接口" → 把假设写成用户需求
✅ Agent 假设："暂按不改变外部接口处理；如有变化请告知"
