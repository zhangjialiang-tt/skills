# Requirement Alignment — 需求对齐

## 目标

在复杂任务开始规划/实现之前，确认 Agent 理解的是用户真正想解决的问题，而不只是用户表面描述的解决办法。

本 Skill 只负责：识别高影响歧义 → 补齐必要信息 → 显式记录假设和边界 → 输出需求对齐摘要。不输出 Plan、不修改代码、不发起冗长访谈。

## 核心原则

1. **先判断是否需要对齐，不默认追问** — 问题清晰时直接跳过
2. **只问阻塞性问题** — 每轮最多 3 个问题，非阻塞信息声明假设继续
3. **先理解目的，再讨论手段** — 区分"想达到的结果"和"提出的实现方式"
4. **最小摩擦** — 问题应具体、能直接改变后续决策，禁止一次性索取完整 PRD
5. **对齐后立即退出** — 输出摘要后交给 Plan 或执行流程

## 触发条件

任务具有一定复杂度，且存在高影响歧义时触发。详见 [references/near-neighbor-routing.md](references/near-neighbor-routing.md) 路由表和 [references/trigger-eval.md](references/trigger-eval.md) 触发评估。

## 对齐级别

### Level 0：直接通过
- 请求清楚、理解唯一、无高影响假设 → 直接进入回答/Plan/执行

### Level 1：轻量对齐
- 主目标清楚，1~2 个歧义影响范围或结果
- 动作：一句话复述 → 最多 2 个问题 → 声明默认假设 → 进入后续阶段

### Level 2：标准对齐
- 多个合理方向，或目标/范围/验收至少一项不明确
- 动作：提取事实/目标/候选理解 → 有限只读探索 → 最多 3 个问题 → 输出摘要

### Level 3：高风险确认
- 可能破坏数据/接口/协议/兼容性/生产环境
- 动作：指出关键分歧 → 2~3 个具体选项 → 确认前不写操作

## 工作流程

### 1. 读取已有上下文
检查：用户请求、本轮已提供信息、项目内 AGENTS.md/README/规格/Issue/Plan/已确认约束。**不得重复询问已明确信息。**

### 2. 拆分请求
区分：**事实**（用户明确提供）/ **目标**（希望发生什么变化）/ **手段**（建议的实现方式）/ **约束**（不能改变的条件）/ **验收**（怎样算完成）/ **假设**（Agent 自行推断）。

注意：用户提出的原因可能只是猜测，不能自动视为事实。

### 3. 判断歧义影响
只保留会影响最终行为、修改范围、外部接口、技术路线、风险等级、验收方式、是否需要 Plan 的歧义。低影响细节不阻塞。

### 4. 必要时有限只读探索
仅在读取少量内容可直接消除歧义时进行：确认模块/接口存在、查看当前行为、判断局部还是跨模块、找测试验收入口。禁止完整调研、形成方案、修改文件、固化 Plan。

### 5. 提出关键问题
优先级：最终想解决的问题 → 哪些行为必须改变/保持 → 范围边界 → 成功标准 → 冲突优先级 → 用户指定方案是否硬约束。

推荐形式：
```
我目前理解你的目标是：<一句话目标>。

这里有两个会改变后续方向的问题：
1. <问题一：给出 A/B/C 选项>
2. <问题二：给出明确边界或验收选择>

其余未说明部分，我暂按以下假设继续：
- <非阻塞假设>
```

### 6. 处理"按你理解做"
- 低风险可回退：列出关键假设并继续
- 中风险：选择最小修改、最保守、最可验证路径
- 高风险/不可逆：仍需确认唯一阻塞性授权问题
- 不把"你决定"解释为允许扩大范围

### 7. 输出需求对齐摘要

模板和示例见 [references/examples.md](references/examples.md)，工件设计规范见 [reports/artifact-design-profile.md](reports/artifact-design-profile.md)。

## 与 Plan 模式的边界

需求对齐回答：为什么做 / 最终要什么 / 什么范围 / 什么不能改变 / 怎样算完成。

Plan 模式回答：修改哪些文件 / 什么技术方案 / 什么顺序 / 如何验证回退 / 如何控制风险。

进入 Plan 模式时，把"需求对齐摘要"作为输入，不重新解释用户目标，不扩大已确认范围。
