# 失败模式清单

执行完成后逐项自检，确保没有踩中这些坑。

## 必须避免

| # | 失败模式 | 后果 | 防御 |
|---|---------|------|------|
| 1 | 对所有复杂任务进行固定问卷 | 用户疲劳，信任下降 | 先判断是否需要对齐，问题清晰时直接跳过 |
| 2 | 追问用户已经说过的信息 | 浪费时间，显得不专业 | 先读取已有上下文，不重复询问 |
| 3 | 把"用户提出的方案"等同于"用户真实目标" | 方案可能不是最优，目标可能未对齐 | 先确认目标，再讨论手段 |
| 4 | 在需求未对齐前输出大段实现计划 | 方向可能错误，返工成本高 | 先输出摘要，再进入 Plan |
| 5 | 用大量技术细节掩盖目标不明确 | 用户更难发现方向问题 | 问题应具体、能直接改变后续决策 |
| 6 | 为显示严谨而要求用户定义所有边界 | 访谈冗长，体验差 | 非阻塞信息声明假设继续 |
| 7 | 把 Agent 假设写成用户确认的需求 | 后续冲突时难以区分来源 | 摘要中明确区分"已确认"和"假设" |
| 8 | 用户回答后再次重复同一组问题 | 用户困惑，体验差 | 对齐后立即退出，后续阶段不重复 |
| 9 | 借需求对齐之名扩大任务范围 | 用户不信任，范围蔓延 | 只问阻塞性问题，不主动扩展 |
| 10 | 将本 Skill 变成 PRD 生成器或 Plan 生成器 | 超出职责，产出不对 | 输出摘要后交给 Plan 模式 |

## 自检清单

- [ ] 是否已判断需要对齐（而非机械触发）？
- [ ] 是否已读取已有上下文，不重复询问？
- [ ] 是否区分了目标、手段、约束、验收、假设？
- [ ] 问题是否都是阻塞性的（≤3 个）？
- [ ] 是否给出了具体选项而非开放性问题？
- [ ] 摘要中是否区分了"用户确认"和"Agent 假设"？
- [ ] 对齐后是否立即退出，不重复访谈？
- [ ] 是否未输出完整技术方案或 Plan？
- [ ] 是否未修改任何文件？
