# 需求对齐示例

## 示例一：无需触发（Level 0）

用户：
```
把 src/foo.py 中的超时时间从 10 秒改为 30 秒，并运行现有单元测试。
```

处理：目标、范围和验收清楚 → 直接执行，不进入需求对齐。

---

## 示例二：轻量对齐（Level 1）

用户：
```
优化这个模块的性能。
```

处理：
```
我理解你希望降低该模块的性能瓶颈，但"性能"可能指吞吐率、延迟或资源占用。

需要确认：
1. 本次优先优化吞吐率、延迟，还是 FPGA 资源占用？
2. 是否允许改变 pipeline latency？

其余部分暂按"不改变外部接口和功能行为"处理。
```

---

## 示例三：目标与手段分离（Level 1）

用户：
```
给这个模块增加一个 FIFO，解决偶发丢帧。
```

处理：
```
我理解你的真实目标是消除偶发丢帧；"增加 FIFO"可能是建议方案，而不一定是硬约束。

需要确认：
1. 必须采用新增 FIFO，还是允许先定位丢帧根因后选择最小修复？
2. 验收是否以连续运行指定帧数且零丢帧为准？
```

---

## 示例四：标准对齐（Level 2）

用户：
```
重构 ir_compress_simple_top，让它支持 1280x1024 和 640x512 两种分辨率。
```

处理：
```
我理解你的目标是让压缩模块支持多分辨率。以下是我的理解和待确认项：

已知事实：
- 当前模块仅支持 1280x1024
- 需要新增 640x512 支持

待确认：
1. 两种分辨率是同时支持（运行时切换）还是编译时参数化？
2. 重构范围：只改 ir_compress_simple_top，还是包括上游 image-rd-cdc 和下游 memctrl？
3. 验收标准：仿真通过即可，还是需要上板验证？

假设（如无异议继续）：
- 不改变模块外部接口（AXI/AXIS 信号列表不变）
- 不改变现有 1280x1024 的功能行为
```

---

## 示例五：高风险确认（Level 3）

用户：
```
把 GVSP 协议从定帧模式改为压缩帧模式，删除定帧相关代码。
```

处理：
```
这是一个高风险操作，涉及协议变更和代码删除。在动手前需要确认几个关键分歧：

1. 定帧模式是否完全废弃？还是保留为 fallback 选项？
   - A. 完全删除，只保留压缩帧
   - B. 保留定帧作为 fallback，通过寄存器配置切换
   - C. 先实现压缩帧，定帧代码暂不删除

2. 压缩帧模式是否已验证通过？
   - 如果尚未验证，建议先验证再删除定帧代码

3. 删除范围：只删 TX 路径，还是包括 RX 路径的发现/注册逻辑？

请确认以上，我再输出需求对齐摘要。
```
