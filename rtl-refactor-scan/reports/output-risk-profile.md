# Output Risk Profile — rtl-refactor-scan

> 此技能主要失败模式与防御策略。内容短小，聚焦可防可控的风险。

## 风险矩阵

| # | 失败模式 | 后果 | 防御 | 严重度 |
|---|---------|------|------|--------|
| 1 | 假设 RTL 文件路径而未确认 | 扫描错误文件 / 漏扫，审计结论失真 | Phase 1 先用 ls/find/rg 确认文件存在，不猜路径 | 高 |
| 2 | 误将 C 类风险标为 A 类 | 过度修改稳定代码，引入新 bug | Gate 3 风险分类复核；C 类只报告不修改 | 高 |
| 3 | 修改改变了模块外部接口 / 协议语义 | 上下游模块失效，难以定位 | Gate 5 检查项（接口不变 / 协议不变 / 不降 coverage） | 严重 |
| 4 | 回归测试失败却屏蔽问题通过 | 真实缺陷被隐藏，上板才暴露 | Gate 6 失败必须分析原因，禁止"屏蔽问题" | 严重 |
| 5 | 未确认即执行修改 | 用户不知情，修改不可控 | Phase 4 无计划不修改；计划需用户确认 | 高 |
| 6 | 大模块一次性扫描 | 超出上下文，维度覆盖不全 | >5000 行建议分模块 / 只扫关键路径 | 中 |
| 7 | 把"快速语法检查"误路由到本 skill | 与 rtl-annotator 职责重叠，产出不对 | frontmatter 排除提示 + trigger-eval.md 边界 #1（Phase 1 确认 quick vs systematic） | 中 |
| 8 | A/B/C 分类标准不统一 | 报告风险等级不可比，计划失焦 | 严格对照 references/risk-classification.md | 中 |
| 9 | 报告缺行号位置 | 用户无法定位问题 | 每个发现点必须记录位置(行号) | 中 |
| 10 | 为精简代码而破坏 EDA 友好性 | 综合/LA 探测不稳定 | 约束：FPGA EDA 友好优先于代码短小 | 中 |
| 11 | 盲目使用 SystemVerilog 语法 | 部分 EDA 工具链不友好 | 优先 Verilog-2001，非必要不升级语法 | 低 |
| 12 | 无 testbench 仍声称"已验证" | 虚假验证结论 | 无 testbench 时 Phase 5 降级为人工确认，明确标注 | 高 |

## 自检清单

执行完成后逐项验证：

- [ ] Phase 1 已确认文件路径、时钟/复位、接口协议（不靠假设）
- [ ] 5 个维度全部扫描，无遗漏
- [ ] 每个发现点含位置(行号) + 分类(A/B/C) + 描述 + 影响
- [ ] A/B/C 分类对照 risk-classification.md，无错级
- [ ] C 类只报告未修改
- [ ] 进入修改前已输出计划并获用户确认
- [ ] 修改未改变外部接口 / 协议 / ready-valid / reset
- [ ] 未删除有用注释、未降 coverage、未屏蔽问题
- [ ] 有 testbench 时回归已 PASS；无 testbench 时明确标注未验证
- [ ] 三份产物（审计/计划/最终）已写入文件
- [ ] 最终报告含剩余风险与下一步建议
