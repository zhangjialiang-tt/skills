# RTL-Refactor-Scan Trigger 评估

用于判断用户请求是否应该触发本 Skill，以及应该按哪种深度模式触发。

## ✅ 应该触发

| # | 用户意图 | 关键词示例 |
|---|----------|------------|
| 1 | RTL 代码评审/审查 | "帮我评审这段 RTL"、"检查下这个 Verilog 代码" |
| 2 | 工程成熟度检查 | "代码成熟度如何"、"工程化检查"、"RTL 质量扫描" |
| 3 | 综合/时序问题诊断 | "综合报错"、"时序不满足"、"上板有问题" |
| 4 | 小步重构请求 | "帮我优化下这段代码"、"重构建议"、"怎么改更好" |
| 5 | EDA 调试友好性 | "LA 探测不到信号"、"debug 不方便" |

## ❌ 不应该触发

| # | 用户意图 | 原因 | 建议替代 |
|---|----------|------|----------|
| 1 | 新功能开发 | 非扫描/重构任务 | 直接编写代码 |
| 2 | 纯 testbench 编写 | 无 RTL 修改 | 使用 testbench 编写技能 |
| 3 | 综合/布局布线执行 | 本 Skill 只做扫描分析 | 使用 EDA 工具 |
| 4 | 纯注释/可读性增强 | 不需要系统性扫描 | 使用 `rtl-annotator` skill |
| 5 | 代码格式化 | 非工程成熟度问题 | 使用代码格式化工具 |

## 🤔 边界情况

| # | 情况 | 处理 |
|---|------|------|
| 1 | 用户要求"检查代码"但未说明范围 | 先询问是快速检查还是系统性扫描（见下方深度模式决策） |
| 2 | 模块过大（>5000 行）| 建议分模块扫描，或只扫描关键路径 |
| 3 | 混合语言（Verilog + SystemVerilog）| 按文件类型分别处理，优先 Verilog-2001 |
| 4 | 无 testbench | 仍然可以扫描，但无法回归验证（Phase 5 降级为人工确认） |

## 扫描深度模式

按用户期望的输出深度，区分三种模式。**Phase 1 启动前必须先确认模式**。

| 模式 | 适用场景 | 典型关键词 | 流程 | 输出物 |
|------|----------|------------|------|--------|
| **快速模式** | 想了解"有没有 A 类阻塞性风险" | "快速看下"、"有没有低级错误"、"先扫一眼" | Phase 1 + Phase 2 简化（仅 A 类） | 简短 A 类风险清单 + 升级提示 |
| **报告模式** | 想拿结构化报告做评审，但不修改 | "做完整扫描"、"输出审计报告"、"工程化评估" | Phase 1 → Phase 3（不出修改计划） | 完整审计报告（含 A/B/C） |
| **完整模式**（默认）| 扫描 + 安全重构 + 回归验证 | "帮我改"、"重构 + 回归"、"完整流程" | Phase 1 → Phase 6 | 审计报告 + 修改计划 + 最终报告 |

**默认模式**：若用户未明确说明深度，按"报告模式"启动；在 Phase 1 末尾向用户确认是否升级到"完整模式"。

## 与近邻 Skill 的梯度决策

`rtl-annotator`（注释增强） ↔ `rtl-refactor-scan` 快速模式 ↔ `rtl-refactor-scan` 报告模式 ↔ `rtl-refactor-scan` 完整模式

| 边界 | 决策信号 |
|------|----------|
| 走 `rtl-annotator` | 用户说"只加注释"、"不改变功能"、"零功能修改" |
| 走本 skill 快速模式 | 用户说"快速"、"低级错误"、"先扫一眼"；且未要求改代码 |
| 走本 skill 报告模式 | 用户说"扫描"、"评审"、"评估"；明确说"不要改代码"或未表态 |
| 走本 skill 完整模式 | 用户说"重构"、"改"、"修"、"优化 + 回归" |

## Trigger 质量检查

触发后，在执行 Phase 1 前必须确认：

1. 用户是否已提供待分析的文件路径？
2. 是否是 Verilog/SystemVerilog 文件（.v / .sv）？
3. **用户期望的输出深度是什么？（快速 / 报告 / 完整）**
4. 用户期望的输出是什么（仅报告 / 报告+计划 / 完整重构）？

如果以上有任何不确定，先与用户确认再启动扫描流程。
