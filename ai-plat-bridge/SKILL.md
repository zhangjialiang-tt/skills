# AI-Plat Bridge — 跨路径工作台桥接

当 WorkBuddy 当前工作目录不在 AI-Plat vault 时，通过本 skill 对工作台进行读写。

## 常量

```text
VAULT = C:\Users\zhangjl\Documents\obsidian-syncthing\AI-Plat\AI-Plat
```

本 skill 中所有路径均相对于 VAULT 根目录。

## 操作模式

### 只读模式

用户要求查询工作台、项目状态、今日日志、任务或记忆时触发。

- 只执行读取，不进行任何写入。
- 不要求本次对话产生工程变化。
- 读取结果直接呈现给用户，不持久化。

### 写入模式

仅在以下情况触发：

- 用户明确要求记录或同步到工作台；
- AI 完成实质性工程工作，并产生可持久化变化。

写入前必须先确认目标文件、读取现有内容、选择同步级别，完成后验证写入结果。

## 目录结构

| 路径 | 用途 |
|------|------|
| `10-Daily/YYYY-MM-DD.md` | 每日动作日志 |
| `20-Tasks/active/*.md` | 进行中的任务卡片 |
| `20-Tasks/done/*.md` | 已完成的任务卡片 |
| `30-Projects/*.md` | 项目文件（含 YAML frontmatter） |
| `40-Prompts/*.md` | 6 个核心 Prompt 模板 |
| `50-Agents/agent-rules.md` | Agent 治理规则 |
| `60-Reviews/review-template.md` | 周复盘模板 |
| `.workbuddy/memory/MEMORY.md` | 项目级长期记忆 |
| `.workbuddy/memory/YYYY-MM-DD.md` | 项目级每日记忆日志 |
| `AGENTS.md` | 仓库级指导文件 |

## 触发与排除

### 触发条件（满足任一即触发）

1. 用户明确要求：记录到工作台、查工作台、同步进展、更新项目文件等
2. AI 完成实质性工程工作后**主动判断**，本次工作至少产生一项可持久化变化：
   - 项目状态发生变化
   - 完成了实际动作（修改、构建、验证通过）
   - 发现了问题的原因
   - 形成了有证据支持的结论
   - 确定了技术决策
   - 改变或解除了阻塞点
   - 明确了下一步动作
   - 推进了任务生命周期（创建、完成、归档）

### 排除条件（任一满足即不触发）

1. AI 当前工作目录已在 AI-Plat vault 内 → 直接按 AGENTS.md 操作，不需桥接
2. 纯聊天、日常问候
3. 信息查询（查文档、查语法、查 API），没有工程动作
4. 未落地的方案讨论、头脑风暴
5. 简单代码阅读、浏览文件
6. 仅格式或注释修改
7. 临时尝试、没有形成工程状态变化的实验

用户明确要求触发时，仍需遵守"只写有实质内容"原则——无可记录内容时直接说明。

## 同步级别选择

按信息价值和工程影响选择最轻的流程，不默认走完整五步。

### 轻量同步

**适用**：小参数调整、局部修复、单次验证通过、微小改进。

操作：
- 追加今日日志（做了什么、推进了哪些项目）
- 按需更新项目文件的当前状态或下一步动作

**不包括**：任务卡片操作、MEMORY.md 更新。

### 标准同步

**适用**：项目状态、目标、阻塞点或关键结论发生变化。

操作：
- 先读取项目文件，再更新相关章节
- 追加今日日志
- 按需更新已有任务卡片的执行记录或下一步动作
- 按需追加每日记忆

**不包括**：创建新任务卡片、归档任务卡片、更新 MEMORY.md。

### 深度同步

**适用**：独立任务、里程碑达成、架构决策、长期约定、完整排查过程。

操作：
- 更新项目文件（当前状态、已验证结论、关键决策、里程碑）
- 创建或归档任务卡片
- 追加今日日志 + 每日记忆
- 判断信息是否具备长期复用价值，是则更新 MEMORY.md

**升级原则**：

- 默认选择轻量同步。
- 产生新证据、新目标、状态变化或阻塞点变化时，升级为标准同步。
- 需要创建或归档任务卡片时，升级为深度同步。
- 涉及里程碑、架构选择、长期约定或跨 session 决策时，升级为深度同步。

## 读取操作

### 查工作台整体状态

读取 `30-Projects/` 下所有 `.md` 文件的 frontmatter，汇总各项目 `status`、当前阶段、当前目标、阻塞点。

### 查具体项目

读取 `30-Projects/<project>.md` 获取完整项目上下文（系统架构、当前快照、工程入口、已验证结论、关键决策、里程碑）。

### 查今日日志

读取 `10-Daily/YYYY-MM-DD.md`（取当前日期）。

### 查活跃任务

列出 `20-Tasks/active/` 下所有 `.md` 文件。

### 查项目记忆

读取 `.workbuddy/memory/MEMORY.md` 获取项目级长期决策和约定。

### 查每日记忆

读取 `.workbuddy/memory/YYYY-MM-DD.md`（取当前日期）。

## 写入操作

### 追加今日日志

文件：`10-Daily/YYYY-MM-DD.md`

如文件不存在，按以下模板创建：

```markdown
# YYYY-MM-DD

## 今天做了什么
- <项目名>：<具体动作>

## 今天推进了哪些项目
- <项目列表>

## 今天遇到的问题

## 今天形成的结论

## 明天最重要的一件事
```

创建时保留固定节标题，但只填写本次能够确认的内容：
- 不使用"无""暂无""待定"等占位条目。
- 不知道明天最重要的一件事时，保持该节为空。
- 不根据上下文擅自推断用户明天的优先事项。

如文件已存在，追加规则：

- 同一天、同一项目、同一工作主题的多次同步，在已有一级条目下追加二级条目，不重复创建语义相同的一级条目。
- 示例：
  ```markdown
  ## 今天做了什么
  - zc29a：调试 ASIC 程序升级功能
    - 完成升级数据发送链路检查
    - 发现响应超时发生在状态轮询阶段
  ```
- 新的独立工作主题可以创建新的一级条目。
- 不重写已存在的历史细节。
- 在 `## 今天推进了哪些项目` 节补充新项目名。

### 更新项目文件

文件：`30-Projects/<project>.md`

仅更新需要变更的节，不覆写无关内容。重点关注：
- `### 当前目标` — 当前阶段最需要推进的事
- `### 当前状态` — 追加日期标注的状态行
- `### 当前阻塞点` — 更新阻碍项
- `### 下一步动作` — 更新待办列表
- `### 已验证结论` — 追加有证据支持的结论
- `## 更新记录` — 追加日期标注的更新行

### 创建任务卡片

文件：`20-Tasks/active/<date>-<slug>.md`

模板：

```markdown
---
created: YYYY-MM-DD
status: active
tags: [task, <project-tag>]
---

# <标题>

## 任务目标

## 背景信息

## 相关路径

## 已知现象

## 约束条件

## 执行记录

## 最终结果

## 下一步动作
```

### 归档任务卡片

将状态改为 `resolved` 或移动到 `20-Tasks/done/`，追加 `## 最终结果` 节。

### 更新项目记忆

文件：`.workbuddy/memory/MEMORY.md`

仅追加长期稳定、会重复影响协作的信息。

写入 MEMORY.md 前，至少满足以下一项：
- 会影响未来多个 session 的决策或操作；
- 能避免后续重复排查或重复犯错；
- 属于稳定的项目约定、架构决策或工作流；
- 在当前项目后续阶段仍会被反复引用；
- 预计三个月后仍有参考价值。

仅描述当天动作、一次性现象或短期进度的信息，不写入 MEMORY.md。

### 追加每日记忆

文件：`.workbuddy/memory/YYYY-MM-DD.md`

追加简洁的工作记录行，格式：`- <项目>：<动作/结论>`。

## 输出风险防御

写入工作台前，必须遵守以下防御规则（按风险优先级）。

### 写入目标确认

写入前必须确认当前工程与工作台项目文件的映射关系。

定位顺序：
1. 根据当前仓库名、路径和项目上下文匹配 `30-Projects/` 中的项目文件。
2. 优先使用项目文件中的工程入口路径、仓库路径或别名字段确认。
3. 无法唯一匹配时，不猜测、不创建新项目文件。列出候选项目并说明无法安全写入的原因。

为提高匹配确定性，项目文件 YAML frontmatter 可包含：

```yaml
aliases: [ZC29A_FPGA, zc29a]
repo_paths:
  - C:\100-Working\102-Working-Prj\zc29a\02\
```

### P0 — 防止覆盖已有内容

- **写入前必须先读取目标文件**。不假设文件为空或内容已知。
- 仅做章节级、局部修改。禁止无必要的整文件重写。
- 编辑前确认目标节头存在；不存在时追加到合理位置而非静默创建重复节。
- 新信息与已有记录冲突时，不静默覆盖。保留旧记录并以如下标注追加变更说明：

  ```markdown
  > [!note]- YYYY-MM-DD 变更
  > <冲突内容和更新原因>
  ```

### P1 — 防止写入错误或未验证结论

- 只有经过验证的结果（仿真通过、编译通过、板级验证、日志确认）才能进入 `已验证结论` 和 MEMORY.md。
- 初步判断、猜测、待验证方向必须明确标记为 `[待验证]`、`[推测]` 或 `[假设]`，不能写成确定事实。
- 不确定是否已修复的问题，写作"现象已不再出现（根因待确认）"而非"已修复"。

### P2 — 防止重复记录

- 写入前检查目标文件中是否已有语义等价的记录。同一结论、同一动作不重复追加。
- 同一天的多次同步应在已有日志条目下追加二级条目，而非创建新的一级条目描述同一件事。

### P3 — 防止漏掉信息

- 形成新结论或决策时，检查是否同时需要更新：项目文件的当前状态 + 今日日志 +（如有长期价值）MEMORY.md。
- 发现阻塞点时，确保项目文件的阻塞点节得到更新。

### 脱敏

- 不写入密钥、密码、Token、客户数据，以及包含敏感账号、客户名称、内部地址或凭据参数的原始日志和路径。
- 必要时脱敏后记录。

## 写后验证

完成写入后，不直接声称"已同步"。必须：

1. 重新读取被修改的目标章节，确认内容已正确落盘。
2. 检查是否出现重复标题、重复条目或格式破坏。
3. 确认未修改无关章节。
4. 只有验证成功后才能向用户报告"已同步"。
5. 写入失败或结果不确定时，明确说明失败项，不声称完成。

验证后输出统一格式：

```text
已同步到工作台：

- 10-Daily/2026-07-17.md
  - 追加"今天做了什么"
- 30-Projects/zc29a.md
  - 更新"当前状态"
  - 更新"下一步动作"

未更新：
- MEMORY.md：本次信息不具备长期复用价值
- 任务卡片：本次不涉及独立任务生命周期
```

## 基础约定

1. **YAML frontmatter**：所有项目/任务文件需含 `created`、`status`、`tags` 字段。
2. **日期格式**：`YYYY-MM-DD`。
3. **中文输出**：所有写入内容使用中文。
4. **最小修改**：只更新与本次工作相关的节，不改变文件其他部分的结构或内容。
5. **只写有实质内容**：不为了"记录"而记录空壳信息。没有值得写的结论就跳过对应节。
